# QueryCreator - 설계 문서

## 개요

Oracle DB 기반 레거시 시스템에서 시스템 담당자에게 반복적으로 들어오는 데이터 확인 문의를 LLM이 자동으로 처리하는 시스템.

Agent 플랫폼 위에서 Python 앱으로 동작하며, LLM이 메타데이터를 참조하여 SQL을 자동 생성/실행하고 결과를 사용자에게 반환한다.

## 배경 및 문제

- 시스템 담당자에게 데이터 확인 문의가 과도하게 발생
- 모든 조회 화면을 만들어줄 수 없는 상황
- MCP/쿼리를 만들어줘도 추가 요청이 계속 발생
- **근본 해결**: LLM이 스스로 쿼리를 생성하여 사용자가 자율적으로 데이터 조회

## 환경

| 항목 | 내용 |
|------|------|
| 플랫폼 | Agent — Python 앱 등록 방식 (서버리스) |
| LLM | Qwen, H-Chat (Sonnet, GPT 5.4, Gemini 3.1) |
| DB | Oracle, SELECT 전용 계정 |
| 테이블 규모 | 스키마별 수십 개, 컬럼 수개~수백 개 |
| 코드값 | 공통코드 테이블로 대부분 커버 |
| 스토어드 펑션 | 정형화, 문서화 가능, 추가 개발 가능성 있음 |
| 외부 서버 | 없음 — Agent가 Python 코드 실행 |

## 메타데이터 4계층 구조

### Layer 1: 업무 사전 (Business Dictionary) — 운영자 필수 등록

업무 용어와 DB 객체 간의 매핑. LLM이 사용자의 자연어 질문을 DB 테이블/펑션으로 연결하는 핵심.

```yaml
tables:
  TB_ORDER:
    업무명: 주문
    설명: "고객 주문 마스터 테이블"
    주요컬럼:
      ORDER_NO: 주문번호
      CUST_CD: 고객코드
      ORDER_DATE: 주문일자

functions:
  F_SUM_PROGRESS:
    업무명: 공정별 진행량 합산
    설명: "주문번호를 넣으면 공정 단계별 중량 합계를 리턴"
    파라미터:
      P_ORDER_NO: 주문번호
    용도: "주문의 공정별 생산량 조회 시 사용"
```

운영자는 주요 테이블/컬럼/펑션만 등록. 나머지 컬럼은 Oracle 딕셔너리에서 자동 보완.

### Layer 2: 물리 메타데이터 — Oracle 딕셔너리에서 자동 수집

- 테이블/컬럼 정보 (ALL_TAB_COLUMNS)
- 행 수 추정치 (ALL_TABLES.NUM_ROWS)
- 인덱스 정보 (ALL_IND_COLUMNS)
- PK/FK 제약조건 (ALL_CONSTRAINTS)
- 컬럼 코멘트 (ALL_COL_COMMENTS)

### Layer 3: 공통코드 매핑 — DB에서 자동 수집

공통코드 테이블에서 코드그룹-코드값-코드명 매핑을 수집하여 LLM에게 제공.

### Layer 4: 운영자 지식 (Knowledge Base) — 선택적 보강

문제가 발생한 후 점진적으로 축적:
- 인덱스 힌트: "이 테이블은 반드시 ORDER_DATE 인덱스 활용"
- 샘플 쿼리: 검증된 쿼리 패턴
- 위험 테이블 표시: "10억건, 반드시 날짜 조건 필수"
- 조인 규칙: "A↔B 조인 시 PLANT_CD + ORDER_NO 복합키"
- 금지 패턴: "LIKE '%검색어%' 금지"

## LLM에게 노출하는 도구 (Tools)

| 도구 | 역할 | 입력 | 출력 |
|------|------|------|------|
| `get_metadata` | 업무 사전 + 물리 메타 + 코드 + 힌트 통합 조회 | 업무 키워드 또는 테이블명 | 관련 테이블/컬럼/코드/펑션/힌트 정보 |
| `execute_query` | SELECT 쿼리 검증 및 실행 | SQL 문자열 | 쿼리 결과 (최대 1000건) |
| `call_function` | 스토어드 펑션 호출 | 펑션명 + 파라미터 | 펑션 리턴값 |

## 쿼리 실행 흐름

```
사용자 질문
    │
    ▼
[Step 1] LLM 의도 파악
    │  사용자가 테이블명 직접 언급? → Step 3로
    ▼
[Step 2] get_metadata 호출
    │  업무 사전에서 키워드 매칭 → 관련 테이블/펑션/힌트 반환
    ▼
[Step 3] LLM이 SQL 생성
    │  펑션이 적합하면 → 펑션 호출 SQL
    │  아니면 → 직접 SELECT 생성
    ▼
[Step 4] 쿼리 검증 (Python)
    │  ✓ SELECT만 허용
    │  ✓ ROWNUM/FETCH FIRST 존재
    │  ✓ 대용량 테이블에 조건절 확인
    │  ✓ 금지 패턴 체크
    │  ✗ 위반 시 → LLM에 수정 요청 (최대 2회)
    ▼
[Step 5] Oracle 실행
    │  타임아웃: 30초 / 결과 행수 제한: 1000건 / 로그 기록
    ▼
[Step 6] LLM이 결과 정리
    │  코드값 → 업무명 변환 / 자연어 응답
    ▼
사용자에게 응답
```

## 안전장치

### 자동 적용 (운영자 개입 불필요)

| 규칙 | 방식 |
|------|------|
| SELECT만 허용 | 쿼리 파싱 후 DML/DDL 차단 |
| ROWNUM/FETCH FIRST 강제 | 검증 레이어에서 없으면 자동 추가 또는 거부 |
| 실행 시간 제한 | Python에서 30초 타임아웃 |
| SELECT * 금지 | 프롬프트 + 검증 레이어 |
| LIKE '%..%' 경고 | 검증 레이어에서 차단 |
| 대용량 테이블 조건 필수 | NUM_ROWS 기반 자동 판단 |

### 점진적 보강 (운영자)

문제 발생 시 운영자가 해당 테이블에 힌트/샘플쿼리/금지패턴 추가 등록.

## 로그 및 관리

### 쿼리 실행 로그

모든 실행 쿼리에 대해 기록:
- 사용자 질문 원문
- 생성된 SQL
- 사용 테이블/조인 패턴
- 실행 시간
- 성공/실패/타임아웃 여부
- 결과 건수

### 느린 쿼리 집계

- 테이블/조인 패턴별 실행시간 집계
- 타임아웃/실패 빈도가 높은 패턴 자동 감지
- 운영자에게 개선 대상 제시

### 관리 채널

- **관리자 페이지 (웹)**: 로그 조회, 느린 쿼리 목록, 힌트/업무사전 등록/수정
- **Claude Code (운영자용)**: 로그 데이터 분석, 개선점 제안

## 컴포넌트 구조

```
querycreator/
├── core/                     # 핵심 기능
│   ├── metadata/
│   │   ├── collector.py      # Oracle 딕셔너리에서 물리 구조 자동 수집
│   │   ├── catalog.py        # 물리 정보 + 업무 사전 통합 관리
│   │   ├── dictionary.py     # 업무 사전 (업무용어 ↔ DB객체 매핑)
│   │   └── knowledge.py      # 운영자 힌트/샘플쿼리/규칙 관리
│   │
│   ├── query/
│   │   ├── validator.py      # 쿼리 검증 (안전규칙 체크)
│   │   ├── executor.py       # Oracle 쿼리 실행 + 타임아웃
│   │   └── formatter.py      # 결과 포맷팅
│   │
│   └── tools/                # LLM에게 노출되는 도구 정의
│       ├── get_metadata.py
│       ├── execute_query.py
│       └── call_function.py
│
├── logging/                  # 로그 수집
│   ├── query_log.py          # 쿼리 실행 이력 기록
│   └── analyzer.py           # 느린 쿼리 집계/패턴 분석
│
├── admin/                    # 관리자 기능 (후순위)
│   ├── web/                  # 관리자 페이지
│   └── reports.py            # 로그 기반 리포트
│
├── config/
│   ├── db_config.py          # Oracle 접속 정보
│   ├── safety_rules.py       # 기본 안전규칙 정의
│   └── schema_config.py      # 대상 스키마 설정
│
├── data/
│   └── dictionaries/         # 업무 사전 YAML 파일
│
└── app.py                    # Agent 등록 진입점
```

## 구현 우선순위

| 순서 | 모듈 | 이유 |
|------|------|------|
| 1 | config/ + data/dictionaries/ | DB 연결 설정 + 업무 사전 구조 정의 |
| 2 | core/metadata/ | 메타데이터 자동 수집 + 업무 사전 통합 — 모든 것의 기반 |
| 3 | core/query/validator.py + executor.py | 안전하게 쿼리 실행 |
| 4 | core/tools/ + app.py | LLM에게 도구 노출, 실제 동작 |
| 5 | logging/query_log.py | 실행 이력 수집 시작 |
| 6 | core/metadata/knowledge.py | 운영자 힌트 등록 기능 |
| 7 | logging/analyzer.py + admin/ | 로그 분석, 관리자 페이지 |

## 기술 스택

- Python 3.11+
- oracledb (Oracle DB 접속 — cx_Oracle 후속, thin 모드로 Oracle Client 불필요)
- PyYAML (업무 사전 파일 관리)
- SQLAlchemy (선택적 — 쿼리 파싱/검증 보조)
- FastAPI (관리자 페이지 — 후순위)
